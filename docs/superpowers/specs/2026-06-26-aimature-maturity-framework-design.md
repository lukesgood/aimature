# AIMature — 바이브 코딩 결과물 프로덕션 성숙도 평가 프레임워크 & 분석기

- 작성일: 2026-06-26
- 상태: 설계 승인 완료 (구현 계획 대기)
- 산출물: 선언적 평가 프레임워크 + 이를 실행하는 자동 코드 분석기(CLI)

## 1. 목적 / 배경

바이브 코딩(AI 주도 빠른 개발)으로 만들어진 애플리케이션은 기능은 동작해도 비기능 요소(보안·안정성·확장성·유지보수성)에서 무너지는 경우가 많다: 시크릿 하드코딩, 에러 처리 누락, 테스트 부재, 확장 불가능한 구조, 여러 AI 세션이 남긴 파편화된 컨벤션 등.

본 프로젝트는 이러한 결과물이 **프로덕션에 활용 가능한지**와 **개발 성숙도**를 객관적으로 평가하기 위한 프레임워크를 정의하고, 실제 리포지토리를 스캔해 점수·증거·권고를 산출하는 분석기를 제공한다.

핵심 설계 원칙: **프레임워크(루브릭)는 선언적 데이터로 정의하고, 분석기는 그것을 실행하는 엔진이다.** 이로써 "프레임워크 문서"와 "자동 분석기"를 동시에 만족시키고, 평가 기준을 코드 수정 없이 조정할 수 있다.

## 2. 평가 프레임워크 모델

### 2.1 평가 축(Pillar)과 기본 가중치

| 축 | 가중치 | 대표 평가 항목(criteria) |
|---|---|---|
| 보안 Security | 30% | 하드코딩된 시크릿/키, 의존성 취약점, 인증·인가 존재, 입력 검증, 주입 방어, CORS/보안 헤더 |
| 안정성 Reliability | 25% | 에러 처리 패턴, 테스트 커버리지, 로깅·모니터링, 헬스체크, 입력 경계 처리 |
| 확장성 Scalability | 20% | 아키텍처 계층 분리, 상태 비저장성, DB/쿼리 효율(N+1 등), 캐싱, 비동기·큐 |
| 유지보수·운영 Maintainability/Ops | 25% | 코드 구조·복잡도·중복, 컨벤션 일관성, 문서/README, CI/CD, 환경설정 관리, 컨테이너화 |

가중치 합계는 100%. 모든 값은 `config/framework.yaml`에서 조정 가능.

### 2.2 점수 산출

각 평가 항목(criterion)은 다음을 산출한다:

- `score`: 0~100
- `evidence`: 근거 목록 (`file:line` + 설명)
- `confidence`: 0~1 (어떤 계층이 측정했는지에 따라 결정)
- `source`: heuristic | tool | llm

축 점수 = 항목 점수의 가중 평균. 전체 점수 = 축 점수의 가중 합산.

### 2.3 성숙도 레벨 매핑

| 레벨 | 점수 구간 | 의미 |
|---|---|---|
| L0 프로토타입 | 0–40 | 데모용. 프로덕션 불가. 치명적 보안/안정성 결함. |
| L1 MVP | 40–60 | 제한된 내부/소규모 사용 가능. 주요 리스크 잔존. |
| L2 베타 | 60–75 | 통제된 프로덕션 가능. 보완 후 정식 배포. |
| L3 프로덕션 레디 | 75–90 | 일반 프로덕션 적합. |
| L4 스케일 레디 | 90–100 | 대규모 트래픽·조직 운영 적합. |

추가 규칙(게이트): 치명적 보안 결함(예: 노출된 라이브 시크릿)이 1건이라도 발견되면 전체 점수와 무관하게 레벨 상한을 L1로 제한한다. 이런 게이트 규칙도 config에 선언한다.

### 2.4 바이브 코딩 특화 신호

일반 코드 품질 도구가 놓치는 AI 생성물 특유의 패턴을 별도 탐지하여 관련 항목에 가산/감점:

- 존재하지 않는 API / 할루시네이션 의존성(설치/해석 불가 import)
- 파일마다 다른 컨벤션(여러 세션 흔적) — 포맷/네이밍/구조 불일치
- 죽은 코드 / 미사용 스캐폴드
- 하드코딩된 예시값(placeholder URL, 더미 키, `localhost` 잔존)
- 인증·검증 통째 누락
- `// TODO`, `FIXME`, placeholder 텍스트 잔존

## 3. 분석기 아키텍처

### 3.1 구현 언어

TypeScript / Node.js. 이유: 바이브 코딩 결과물 다수가 JS/TS이며, 외부 도구 어댑터와 LLM SDK 연동이 자연스럽다. JS/TS는 심화 분석, 타 스택(Python 등)은 언어 무관 정적 휴리스틱 + 외부 도구 호출로 커버한다.

### 3.2 3계층 측정 파이프라인

```
repo 경로
  → Layer 1 정적 휴리스틱 (항상 실행, 의존성 0)
       파일트리 스캔 · 정규식/패턴 · 메타데이터
       (테스트 디렉터리, CI 설정, .env 처리, 락파일, README, 시크릿 패턴, 컨벤션 일관성)
  → Layer 2 외부 도구 어댑터 (설치돼 있으면 실행, 없으면 생략)
       eslint · npm/pip audit · semgrep · coverage · 복잡도 도구
       각 어댑터가 결과를 공통 Finding 스키마로 정규화
  → Layer 3 LLM 보조 분석 (API 키 있으면 실행, 선택)
       아키텍처 분리 · 인증 누락 · 에러 처리 품질 등 정성 판단을 루브릭 프롬프트로 점수화
  → Scoring Engine: finding 병합 → 항목별 점수 → 축별 가중합 → 전체 점수 → 성숙도 레벨
  → Reporters: 터미널 요약 · JSON · Markdown/HTML 리포트
```

각 계층은 공통 `Finding` 스키마로 출력한다. 계층이 없으면(도구 미설치/키 없음) 우아하게 생략하고 해당 항목 신뢰도를 낮춘다. 같은 항목을 여러 계층이 측정하면 신뢰도 가중으로 병합한다(LLM·tool·heuristic 순으로 신뢰도 차등).

기본 LLM 모델: Claude(최신 Claude 모델 우선). 키 미설정 시 Layer 3 생략.

### 3.3 모듈 구조

| 모듈 | 책임 | 의존 |
|---|---|---|
| `config/framework.yaml` | 선언적 루브릭: 축·항목·가중치·임계값·레벨 구간·게이트 규칙. 프레임워크의 단일 진실원천 | 없음 |
| `core/` | 타입(`Criterion`, `Finding`, `Report`), 항목 레지스트리, 스코어링 엔진, 레벨 매퍼 | config |
| `collectors/` | Layer 1 정적 휴리스틱 (파일트리·정규식·메타데이터) | core |
| `adapters/` | Layer 2 외부 도구 래퍼 (각 도구 → Finding 정규화, 미설치 시 skip) | core |
| `llm/` | Layer 3 LLM 분석 (코드 샘플링 + 루브릭 프롬프트 + 응답→점수) | core |
| `report/` | 터미널/JSON/Markdown(HTML) 렌더러 | core |
| `cli/` | 진입점 | 전부 |

각 모듈은 단일 책임을 갖고 공통 인터페이스(`Finding`)로만 통신하여 독립적으로 테스트 가능하다.

### 3.4 공통 데이터 스키마 (개념)

```ts
interface Finding {
  criterionId: string;       // 어느 평가 항목에 기여하는지
  score: number;             // 0~100
  evidence: { file: string; line?: number; note: string }[];
  confidence: number;        // 0~1
  source: 'heuristic' | 'tool' | 'llm';
}

interface Report {
  overallScore: number;
  level: 'L0' | 'L1' | 'L2' | 'L3' | 'L4';
  pillars: { id: string; score: number; criteria: CriterionResult[] }[];
  uncovered: string[];       // 측정 불가 항목 (은폐 금지)
  recommendations: string[];
}
```

### 3.5 CLI 인터페이스

CLI 명령어는 `aim`을 사용한다 (패키지/프레임워크명은 AIMature 유지).

```
aim scan <path> [options]
  --llm              Layer 3 LLM 분석 활성화 (기본: 키 있으면 자동)
  --no-tools         Layer 2 외부 도구 생략
  --format <type>    terminal | json | markdown | html  (기본: terminal)
  --out <file>       리포트 파일 출력 경로
  --config <file>    커스텀 framework.yaml 경로
```

### 3.6 데이터 흐름 요약

`scan <repo>` → 3계층이 각자 `Finding[]` 생성 → 엔진이 항목별로 병합·점수화 → 축 가중합 → 성숙도 레벨 → 리포트(증거·권고 포함).

## 4. 에러 처리 전략

- 어댑터/LLM 실패는 전체 중단이 아니라 "해당 항목 미측정 + 신뢰도 하향"으로 흡수한다.
- 리포트에 "측정 불가 항목(`uncovered`)"을 명시한다. 측정하지 못한 것을 측정한 것처럼 보이게 하지 않는다.
- 다수 항목이 미측정이면 전체 신뢰도 경고를 리포트 상단에 표시한다.

## 5. 테스트 전략

- 픽스처 리포지토리: 의도적으로 `safe` / `vulnerable` / `mixed` 샘플 디렉터리를 두고 기대 점수 범위를 명시.
- 각 collector·adapter·llm 모듈을 단위 테스트(어댑터는 도구 출력 정규화 로직을 모킹).
- 스코어링 엔진은 골든 리포트로 회귀 테스트.
- 게이트 규칙(치명적 보안 결함 → 레벨 상한) 전용 테스트.

## 6. 범위 밖 (YAGNI)

- 웹 대시보드 UI / 서버 — 1차 범위는 CLI + 파일 리포트.
- 실시간 CI 통합/배지 — 초기 구현 후 확장 항목.
- JS/TS·Python 외 언어의 심화 플러그인 — 초기엔 언어 무관 휴리스틱 + 외부 도구로 커버.

## 7. 결정 사항 요약

- 산출물: 자동 코드 분석기 (선언적 프레임워크를 실행)
- 평가 축: 보안 / 안정성 / 확장성 / 유지보수·운영 준비성 (4개)
- 측정: 정적 휴리스틱 + 외부 도구 + LLM 보조 (3계층 결합)
- 구현 언어: TypeScript / Node.js (JS/TS 1순위, 타 스택은 휴리스틱·도구로 커버)
